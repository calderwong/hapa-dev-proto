## 2. Strategic Architecture

### The "Overlay" Pattern (Confirmed)
We will utilize the **React Three Fiber (R3F)** stack you already have installed. The architecture is a "Ghost Layer" that sits on top of the DOM.

*   **Layer 0 (DOM)**: The existing React App (Sidebar, Grids, `CardLibrary`).
*   **Layer 1 (The Void)**: A full-screen `<Canvas>` with `pointer-events: none`.
*   **Layer 2 (The Artifacts)**: 3D Meshes inside the Canvas with `pointer-events: auto`.

### The Data Mechanics (Cultivation Focus)
**Problem**: If we mix "Visual Coordinates" (X, Y, Z, Rotation) directly into the persistent `HapaCard` data, we pollute the database with transient state.
**Solution**: We create a **`BattleContext` (or Store)**.
*   **Persistence**: `HapaCard` (Data, Stats, Content) lives in the DB/FileSystem.
*   **Presence**: `BattleEntity` (Position, Velocity, Status) lives in the ephemeral `BattleStore`.
*   **Link**: A `BattleEntity` holds a `referenceId` to the `HapaCard`.

This keeps our data clean. When a "Battle" ends, we can discard the positions but keep the results (XP, modifications).

## 4. UI/UX Design "Bruce Lee" Style

**"Simplicity is the key to brilliance."**

1.  **No "Edit Mode" Toggle**: The 3D layer should *always* be present but subtle. It wakes up when you drag.
2.  **Visual Clarity**:
    *   **Idle**: Cards are semi-transparent holograms (glassy).
    *   **Active**: Cards become solid and opaque when hovered/selected.
3.  **The "Anchor"**: When a card is influencing the system, a literal beam of light connects it to the relevant UI section (e.g., a card modifying "Chat" floats near the chat input).
